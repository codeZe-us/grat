Grat handles the complexity of fee sponsorship so you don't have to. It is built on three core pillars: **Fee-Bump Transactions**, **Channel Pools**, and **Atomic Sequence Tracking**.

## Fee-Bump Transactions (CAP-0015)

At the heart of Grat is a native Stellar feature called **Fee-Bump Transactions**. 

A standard transaction has one signer who pays the fee. A Fee-Bump transaction wraps that transaction in an outer envelope. The outer signer (the Grat Relay) pays the transaction fee, while the inner transaction retains its original sequence number and source account signatures.

## Channel Pools

Submitting many transactions in parallel from a single account on Stellar is difficult because of strict sequence number increments. 

Grat solves this by maintaining a **pool of Channel Accounts**. When a request comes in, the relay:
1. Picks an available account from the pool.
2. Uses that account as the fee-payer for the Fee-Bump.
3. Automatically increments and manages the sequence numbers for that specific channel.

## Protocol 26 Support

Grat tracks Stellar protocol upgrades and handles protocol-specific transaction behavior. With Protocol 26 (Yardstick), the relay correctly processes transactions using new Soroban host functions (checked integer arithmetic, BN254 cryptographic operations, precise TTL extensions) and handles frozen ledger entry rejections (CAP-0077) with a distinct error response rather than a generic failure.
